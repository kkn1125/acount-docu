# PRD: 자산(보유 자산) 기입 및 표시

## 1. 기획 요약 및 배경

### 1.1 배경
- 사용자 요청: "현재 가지고 있는, 보유하고 있는 자산을 기입할 수 있어야 하고, 그래야 수중에 얼마가 있는지 계산될 것 같다."
- 서버에는 이미 **Account** 모델이 있으며 **balance**(Decimal) 필드로 계정별 잔액을 저장할 수 있는 구조다.
- 현재 **GET /api/accounts**로 계정 목록만 조회 가능하며, **잔액을 수정하는 API는 없다.** 클라이언트에도 잔액 기입·수정 UI가 없다.

### 1.2 목표
1. **계정별 현재 잔액을 기입·수정**할 수 있게 한다.
2. **총 자산**(모든 계정 잔액 합계)을 사용자가 볼 수 있게 한다.
3. (선택) 대시보드 또는 자산 전용 화면에서 "수중 자금" 요약으로 노출한다.

### 1.3 제약·가정
- **데모 유저(demo@local)** 기준으로만 동기화해도 된다.
- 잔액은 **사용자가 직접 입력하는 "현재 잔액"** 방식으로 두고, 거래 발생 시 자동으로 잔액을 증감하는 로직은 **이번 범위에 포함하지 않는다**(추후 확장 가능).
- 계정은 시드/설정에서 이미 추가된 것을 사용하며, "계정 추가"는 카테고리 관리처럼 별도 기능이 있다고 가정한다. 이번에는 **기존 계정의 잔액만 기입·수정**하는 데 집중한다.

## 2. 기능 범위 요약

| # | 항목 | 내용 | 비고 |
|---|------|------|------|
| 1 | **계정 잔액 조회** | GET /api/accounts 응답에 `balance` 포함. 클라이언트 타입·매핑에 balance 반영. | 이미 서버는 balance 반환 가능, 클라이언트에서 수신·표시만 추가 |
| 2 | **계정 잔액 수정 API** | PUT /api/accounts/:id 또는 PATCH, body에 `balance`(숫자/문자열). 데모 유저 소유 계정만 수정 가능. | 신규 구현 |
| 3 | **잔액 기입/수정 UI** | 계정 목록(또는 자산 화면)에서 계정별 잔액 입력·저장. 유효성(숫자, 소수 등) 처리. | 신규 또는 기존 자산/설정 화면 확장 |
| 4 | **총 자산 표시** | 모든 계정 balance 합계를 "총 자산"으로 계산·표시. 노출 위치: 자산 전용 화면 필수, 대시보드 등은 선택. | 계산 로직 + UI 컴포넌트 |

## 3. PRD 초안

### 3.1 사용자 스토리
- **US1**: 나는 계정(현금, 은행, 카드 등)별로 **현재 잔액을 입력·수정**할 수 있어서, 실제 보유 자산을 앱에 반영하고 싶다.
- **US2**: 나는 **모든 계정 잔액의 합계(총 자산)**를 한눈에 보고, 수중 자금이 얼마인지 파악하고 싶다.

### 3.2 수용 기준
1. **AC1**: GET /api/accounts 응답에 각 계정의 `balance`가 포함되며, 클라이언트에서 계정 목록 조회 시 잔액이 표시된다.
2. **AC2**: 계정 잔액을 변경하는 API(PUT 또는 PATCH /api/accounts/:id, body: { balance })가 존재하며, 데모 유저 소유의 계정에 대해서만 성공적으로 동작한다.
3. **AC3**: 클라이언트에서 계정별 잔액을 입력·수정할 수 있는 UI가 있으며, 저장 시 위 API를 호출해 서버에 반영되고 목록/총 자산이 갱신된다.
4. **AC4**: 총 자산(모든 계정 balance 합계)이 계산되어 사용자에게 표시되며, 표시 위치는 최소한 "자산" 관련 화면에서 노출된다.
5. **AC5**: 잔액 입력 시 유효한 숫자만 허용되며(음수 허용 여부는 정책에 따라 정의), 잘못된 입력에 대한 피드백이 있다.

## 4. QA 체크리스트

구현 완료 후 검증 시 아래 항목을 확인한다.

- [ ] **잔액 수정 API 호출**: PUT/PATCH /api/accounts/:id에 `balance`를 포함해 요청 시 200 및 수정된 계정 객체가 반환되고, DB에 반영된다.
- [ ] **잔액 입력 유효성**: 클라이언트에서 숫자가 아닌 값, 또는 정책상 허용하지 않는 값(예: 음수) 입력 시 저장이 막히거나 에러 메시지가 표시된다.
- [ ] **총 자산 합계 정확성**: 계정이 2개 이상일 때 총 자산 = 각 계정 balance 합계와 일치한다(반올림/Decimal 표시 정책 일치).
- [ ] **화면 반영**: 잔액 수정 저장 후 목록/총 자산이 즉시 갱신되거나, 새로고침 시 서버에서 조회한 값으로 표시된다.
- [ ] **데모 유저 스코프**: 다른 유저의 계정 id로 수정 요청 시 403/404 등으로 거부된다(데모만 사용 시 데모 유저 계정만 수정 가능한지 확인).
- [ ] **GET 응답에 balance 포함**: GET /api/accounts 응답 각 항목에 balance 필드가 포함되고, 클라이언트 타입/매핑에 balance가 있어 화면에 표시된다.
- [ ] **자산 화면 또는 대시보드 노출**: 총 자산이 최소 한 곳(자산 화면 또는 대시보드)에서 "수중 자금" 요약으로 노출된다.
- [ ] **접근성/표시**: 잔액·총 자산이 금액용 디자인 토큰(예: `--text-amount-*`, tabular-nums)으로 읽기 쉽게 표시된다.

## 5. 구현 시 참고 사항(비기능)

- **IA**: 마스터 스킬의 "자산(Accounts)" — 계좌 목록, 총 자산/총 부채/순자산, 계좌별 잔액 히스토리. 이번 범위는 "계좌별 현재 잔액 기입·수정"과 "총 자산 표시"에 해당한다.
- **Decimal 처리**: 서버 balance는 Prisma Decimal. API에서는 문자열 또는 숫자로 주고받고, 클라이언트에서는 합계 계산 시 정수/소수 처리 정책을 통일한다.
- **선택 확장**: 대시보드 히어로 영역 또는 요약 카드에 "총 자산" 노출 시, 마스터 스킬의 대시보드 정보 위계(잔여 예산 1순위 등)와 충돌하지 않도록 보조 정보로 배치한다.
