# PRD: 카테고리 서버-클라이언트 정합성 및 CRUD

## 1. 개요

### 1.1 목적
- **서버를 카테고리의 단일 진실 공급원(SSOT)**으로 하고, 클라이언트는 서버에서 내려준 카테고리만 사용한다.
- 카테고리에 대한 **생성/수정/삭제 API**를 추가하고, 클라이언트에 카테고리 관리 UI 및 API 연동을 구현한다.
- 기존 클라이언트 로컬 상수(`CATEGORY_LABEL_MAP`, `CATEGORY_IDS` 등) 기반 목록/선택/표시를 **서버 응답 기반**으로 전면 전환한다.

### 1.2 현재 상태 요약 (문제 정의)

| 구분 | 현재 상태 | 문제 |
|------|-----------|------|
| **서버** | 카테고리 DB 관리, seed로 데모 유저 수입/지출 기본 카테고리 생성(id=UUID, name=월급·식비 등). `GET /api/categories?type=INCOME\|EXPENSE` 만 존재. | 생성/수정/삭제 API 없음. |
| **클라이언트** | `categoryAccount.ts`에 로컬 상수: `CATEGORY_LABEL_MAP`(영문 id→한글), `CATEGORY_COLOR_MAP`, `CATEGORY_IDS`. API 호출 없음. | 목록/필터/입력 폼이 전부 상수 기반. |
| **불일치** | 서버 id=UUID, 이름=한글. 클라이언트 id='food','salary' 등. | 거래 생성 시 `categoryId: 'food'` 전송 시 서버 FK 검증 실패. 필터/라벨에서 UUID 오면 `CATEGORY_LABEL_MAP[UUID]` undefined → UUID 노출 또는 fallback만 동작. |

## 2. 목표 사용자 및 성공 기준

### 2.1 목표 사용자
- 가계부 앱을 사용하는 **데모 유저**(또는 추후 인증 도입 시 해당 유저).
- 카테고리 목록 조회·선택(거래 입력/필터/대시/예산)과 **카테고리 추가·수정·삭제**를 수행하는 사용자.

### 2.2 성공 기준
- **정합성**: 모든 화면(QuickAdd, 거래 목록, 필터, 대시보드, 예산)에서 카테고리 **이름·id**가 서버에서 내려준 값과 일치하여 표시된다.
- **CRUD**: 서버에 카테고리 생성(POST)/수정(PUT)/삭제(DELETE) API가 존재하고, 클라이언트에서 카테고리 관리(목록·추가·수정·삭제)가 가능하다.
- **거래·예산 연동**: 거래 생성/수정 시 **서버에 존재하는 categoryId(UUID)**만 선택·전송 가능하며, 예산·대시보드·필터는 동일한 서버 카테고리 데이터를 사용한다.
- **로컬 상수 제거**: 카테고리 목록/선택/표시에 사용하던 `CATEGORY_LABEL_MAP`, `CATEGORY_IDS`(및 이에 의존하는 카테고리 선택 UI)는 제거되고, 서버 응답(또는 클라이언트 캐시)으로 대체된다. (계정·거래 타입 라벨 등 **다른 상수는 유지 가능**.)

## 3. 기능 요구사항

### 3.1 카테고리 목록 조회 (기존 확장)
| 항목 | 요구사항 |
|------|----------|
| API | `GET /api/categories?type=INCOME\|EXPENSE` (기존 유지). 쿼리 없으면 전체 타입 반환. |
| 응답 | `{ id: string(UUID), userId: string, name: string, type: TransactionType, parentId?: string, createdAt, updatedAt }[]` |
| 인증/스코프 | 현재와 동일하게 데모 유저 기준(또는 추후 userId 스코핑). |
| 완료 조건 | 200 + 배열. type 필터 시 해당 타입만 반환. |

### 3.2 카테고리 타입별 필터
| 항목 | 요구사항 |
|------|----------|
| 클라이언트 | 거래 타입(수입/지출)에 따라 조회한 카테고리만 사용. 수입 입력 시 INCOME, 지출 입력 시 EXPENSE. |
| 서버 | `type` 쿼리로 필터링(기존 동작 유지). |
| 완료 조건 | QuickAdd/필터/거래 폼에서 타입에 맞는 카테고리만 노출. |

### 3.3 카테고리 생성
| 항목 | 요구사항 |
|------|----------|
| API | `POST /api/categories` body: `{ name: string, type: INCOME\|EXPENSE, parentId?: string }` |
| 검증 | name 필수, type은 enum 값만 허용. parentId는 존재하는 카테고리 id 또는 null. |
| 응답 | 201 + 생성된 카테고리 객체. |
| 완료 조건 | 생성 후 목록 재조회 또는 응답으로 UI 갱신. |

### 3.4 카테고리 수정
| 항목 | 요구사항 |
|------|----------|
| API | `PUT /api/categories/:id` body: `{ name?: string, type?: INCOME\|EXPENSE, parentId?: string | null }` |
| 검증 | id는 해당 유저 소유 카테고리. name/type 변경 시 중복 등 정책 적용(선택). |
| 응답 | 200 + 수정된 카테고리 객체. |
| 완료 조건 | 수정 후 목록/캐시 갱신, 거래·예산에서 해당 카테고리 이름이 새 이름으로 표시. |

### 3.5 카테고리 삭제
| 항목 | 요구사항 |
|------|----------|
| API | `DELETE /api/categories/:id` |
| 정책 | **선택 정책**(구현 시 하나 확정): (A) 해당 카테고리를 사용하는 거래/예산이 있으면 삭제 거부(409 등). (B) 삭제 시 해당 거래의 categoryId를 지정한 기본 카테고리로 변경 후 삭제. (C) 소프트 삭제(예: isArchived). PRD에서는 **(A) 거부**를 기본 가정. |
| 응답 | 204 No Content 또는 200. 사용 중이면 409 + 메시지. |
| 완료 조건 | 삭제 성공 시 목록에서 제거. 삭제 거부 시 사용자에게 안내. |

### 3.6 거래/예산/대시보드에서의 사용 일원화
| 항목 | 요구사항 |
|------|----------|
| 거래 생성/수정 | 선택 가능한 카테고리 = 서버에서 조회한 카테고리만. 전송하는 `categoryId`는 서버에 존재하는 UUID. |
| 거래 목록/상세 | 표시 시 서버에서 내려준 `category.id`, `category.name` 사용(이미 API가 category 포함 시 동일). |
| 필터(FilterBar) | 카테고리 드롭다운 옵션 = 서버 카테고리 목록(타입 필터 연동). 표시 텍스트 = `category.name`. |
| QuickAddSheet / TransactionCreateModal / TransactionEditModal | 카테고리 선택 옵션 = 서버 카테고리. id=UUID, 라벨=name. |
| 대시보드/예산/Summary | 이미 서버에서 categoryId·categoryName 내려주면 그대로 사용. 클라이언트에서 로컬 맵으로 덮어쓰지 않음. |
| 색상/아이콘 | 현재 스키마에 color/icon 없음. **가정**: 이름/타입만으로 표시. 색상은 클라이언트에서 타입(수입/지출) 또는 id 기반 해시/고정 팔레트로 파생 가능(선택). |

## 4. 비기능 요구사항

### 4.1 일관된 표시
- 앱 전역에서 같은 카테고리는 **동일한 이름**으로 표시된다(서버 name 필드 기준).
- 카테고리 id(UUID)는 API·내부 상태에서만 사용하고, 사용자에게 “카테고리 이름”으로만 노출한다.

### 4.2 에러 처리
- API 4xx/5xx 시 클라이언트에서 사용자에게 메시지 표시(토스트/스낵바 등).
- 거래 생성/수정 시 카테고리 미존재(404/400): “선택한 카테고리를 찾을 수 없습니다. 목록을 새로고침해 주세요.” 등 안내.
- 카테고리 삭제 거부(사용 중): “이 카테고리를 사용하는 거래 또는 예산이 있어 삭제할 수 없습니다.” 등 안내.

### 4.3 클라이언트 캐시/동기화
- 카테고리 목록은 **필요 시점에 조회**(예: QuickAdd 오픈, 설정 화면 진입) 또는 **전역 스토어/캐시**에 저장 후 재사용.
- 카테고리 CRUD 성공 시 목록 캐시 무효화 또는 해당 항목만 갱신하여, 이후 거래 입력/필터에서 즉시 반영.

## 5. 제약 및 가정

### 5.1 제약
- **인증**: 현재는 데모 유저 단일. 추후 다중 유저 시 `userId` 스코핑은 기존 패턴 유지.
- **스키마**: Category 모델은 `id, userId, name, type, parentId` 기준. color/icon 등은 본 범위에서 추가하지 않음(추후 확장 가능).
- **계층 구조**: `parentId`는 스키마에 있으나, MVP에서는 **평면 목록**만 지원해도 됨(계층 UI/정렬은 Nice-to-have).

### 5.2 가정
- seed에 의해 데모 유저의 기본 카테고리(수입/지출)가 이미 존재한다.
- 거래/예산 API는 기존처럼 `categoryId`(UUID)를 사용하며, 카테고리 존재 여부는 FK 또는 명시적 조회로 검증한다.
- **ACCOUNT_LABEL_MAP**, **TRANSACTION_TYPE_LABEL_MAP** 등 카테고리 이외의 상수는 그대로 두고, 제거 대상은 **카테고리 관련 상수 및 그를 사용하는 카테고리 선택/표시 로직**만 해당한다.

# QA 체크리스트: 카테고리 서버-클라이언트 정합성 및 CRUD

## A. API — 카테고리 조회

- [ ] **A1** `GET /api/categories` (type 없음): 200, 응답이 배열이며, 데모 유저의 수입·지출 카테고리가 모두 포함된다.
- [ ] **A2** `GET /api/categories?type=INCOME`: 200, 모든 항목의 `type`이 `INCOME`이다.
- [ ] **A3** `GET /api/categories?type=EXPENSE`: 200, 모든 항목의 `type`이 `EXPENSE`이다.
- [ ] **A4** 각 항목에 `id`(UUID 형식), `name`, `type`, `userId` 등 기대 필드가 포함된다.

---

## B. API — 카테고리 생성

- [ ] **B1** `POST /api/categories` body `{ "name": "테스트카테고리", "type": "EXPENSE" }`: 201, 생성된 객체에 `name`, `type`, `id`가 포함된다.
- [ ] **B2** `name` 누락 또는 `type` 잘못된 값: 400 등 적절한 에러와 메시지.
- [ ] **B3** 생성 후 `GET /api/categories` 호출 시 방금 생성한 카테고리가 목록에 포함된다.

---

## C. API — 카테고리 수정

- [ ] **C1** `PUT /api/categories/:id` body `{ "name": "이름변경" }`: 200, 해당 카테고리 `name`이 변경된 값으로 반환된다.
- [ ] **C2** 존재하지 않는 `id`: 404.
- [ ] **C3** 수정 후 목록 조회 시 변경된 이름이 반영된다.

---

## D. API — 카테고리 삭제

- [ ] **D1** 어떤 거래/예산도 사용하지 않는 카테고리로 `DELETE /api/categories/:id`: 204 또는 200.
- [ ] **D2** 삭제 후 `GET /api/categories`에서 해당 카테고리가 사라진다.
- [ ] **D3** (정책 A: 거부 시) 해당 카테고리를 사용하는 거래가 있을 때 `DELETE`: 409(또는 400) 및 에러 메시지. 삭제되지 않는다.
- [ ] **D4** 존재하지 않는 `id`로 삭제: 404.

---

## E. 거래 API와의 연동

- [ ] **E1** `POST /api/transactions` 시 `categoryId`에 서버에 존재하는 카테고리 UUID 사용: 201, 거래 생성 성공.
- [ ] **E2** `categoryId`에 존재하지 않는 UUID 또는 'food' 같은 로컬 id 전송: 400/404 등 검증 실패(Prisma FK 또는 명시적 검증).
- [ ] **E3** `PUT /api/transactions/:id`로 `categoryId`를 다른 유효한 UUID로 변경: 200, 해당 거래의 카테고리가 변경된다.
- [ ] **E4** 거래 목록 GET 응답에 `category.id`, `category.name`이 포함되어 있으며, 이름이 서버 카테고리와 일치한다.

- [ ] **F1** QuickAddSheet(또는 거래 입력 바텀시트) 오픈 시 카테고리 옵션이 **서버에서 조회한 목록**으로 표시된다(로컬 상수 아님).
- [ ] **F2** 수입/지출 타입 선택 시 해당 타입(INCOME/EXPENSE)의 카테고리만 노출된다.
- [ ] **F3** 선택 시 전송되는 값이 서버 카테고리 **id(UUID)**이다.
- [ ] **F4** TransactionCreateModal / TransactionEditModal에서도 서버 카테고리 목록을 사용하고, 저장 시 서버 categoryId가 전송된다.
- [ ] **F5** FilterBar(또는 내역 필터)의 카테고리 드롭다운 옵션이 서버 카테고리 목록이며, 표시 텍스트는 `category.name`이다. 선택 시 필터에 UUID가 사용된다.
- [ ] **F6** 거래 목록/아이템에서 카테고리 이름이 서버에서 내려준 이름으로 표시된다(UUID가 그대로 노출되지 않음).

- [ ] **H1** 카테고리 관리 화면 또는 모달에서 `GET /api/categories` 목록이 표시된다.
- [ ] **H2** 새 카테고리 추가 시 `POST /api/categories` 호출 후 목록에 반영된다.

## I. 로컬 상수 제거 및 일관성

- [ ] **I1** 카테고리 목록/선택/표시에 `CATEGORY_LABEL_MAP`, `CATEGORY_IDS`를 사용하는 코드가 제거되었거나, 서버 데이터가 없을 때만 fallback으로만 사용된다.
- [ ] **I2** `CATEGORY_COLOR_MAP` 등 카테고리 전용 상수도 서버 기반 표시와 충돌하지 않도록 정리되었다(색상은 타입/해시 등으로 파생 가능).
- [ ] **I3** 거래 생성 플로우에서 “서버 카테고리 id만 선택 가능”한지 end-to-end로 확인된다(예: 서버 id로 저장 후 목록에 올바른 이름 표시).

---

## J. 에러 및 경계

- [ ] **J1** API 오류(500, 네트워크 오류) 시 사용자에게 에러 메시지 또는 토스트가 표시된다.
- [ ] **J2** 카테고리 목록 로딩 실패 시 QuickAdd/필터에서 적절한 fallback 또는 재시도 안내가 있다.
- [ ] **J3** 삭제 거부(사용 중인 카테고리) 시 안내 문구가 명확하다.

## 체크리스트 요약 (실행 시 활용)

| 구역 | 항목 수 | 핵심 확인 |
|------|--------|-----------|
| A. 조회 API | 4 | type 필터, 배열·필드 |
| B. 생성 API | 3 | 201, 검증, 목록 반영 |
| C. 수정 API | 3 | 200, 404, 목록 반영 |
| D. 삭제 API | 4 | 204, 409(사용 중), 404 |
| E. 거래 연동 | 4 | 유효/무효 categoryId, 응답 category |
| F. 클라이언트 목록/선택 | 6 | QuickAdd·필터·모달 서버 연동 |
| G. 대시/예산/Summary | 3 | 서버 이름 일치 |
| H. 카테고리 관리 UI | 5 | CRUD 플로우·반영 |
| I. 로컬 상수 제거 | 3 | 상수 제거·일관성 |
| J. 에러/경계 | 3 | 오류 표시·삭제 거부 안내 |

**총 38항목.** 배포 전 회귀 시 A~E는 API 단위, F~J는 클라이언트/통합 테스트로 나누어 실행할 수 있다.
