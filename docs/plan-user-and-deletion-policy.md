# User 화면 반영·로그인·삭제 정책 계획

## 1. 현재 상태

### 1-1. User 모델 (DB)
- **스키마**: `User` (id, email, name, currency, locale, timezone, createdAt, updatedAt)
- **역할**: 모든 계정·카테고리·거래·예산이 `userId`로 연결됨
- **실제 사용**: 서버에서 **고정 이메일(`demo@local`) 하나**만 조회해 사용. 로그인·세션 없음.

### 1-2. 프론트엔드
- User 관련 API·화면 **없음**. 설정 페이지에 "통화, 로케일" 언급만 있고, 실제로는 DB의 User 값(이름, 통화 등)을 읽거나 수정하지 않음.
- 따라서 **DB에 User가 있어도 화면에는 전혀 반영되지 않은 상태**가 맞음.

### 1-3. User가 “로그인 유저”인지
- 지금 구조는 **단일 데모 유저** 가정이므로, “로그인할 때 쓰는 그 User”와 **개념적으로는 같지만**, 실제로는 **로그인 기능이 없음**.
- 로그인을 넣으면: 여러 사용자 구분, 세션/토큰, 보안 등이 필요해짐.

---

## 2. 목표 정리

| 항목 | 목표 |
|------|------|
| User 화면 반영 | DB의 User(이름, 통화, 로케일 등)를 **설정 등에서 조회·표시·수정** 가능하게 함. |
| User 관리 | “현재 사용 중인 유저 1명”의 **프로필 관리**가 되도록 함. (최소: 조회/수정 API + 설정 UI) |
| 로그인 | User를 “로그인 유저”로 쓸 경우 **로그인 기능**이 필요함. 단계를 나눠 도입 가능. |
| 삭제 정책 | 상위(카테고리·계정 등) 삭제 시 **연관 데이터(거래/수입·지출 이력)는 유지**. nullable 또는 **다른 항목으로 재할당 후 삭제** 방식 + 안내 문구. |

---

## 3. User 화면 반영 제안

### 3-1. API
- **GET /api/user** (또는 `/api/me`)
  - 현재 데모 유저 1명 정보 반환: `id`, `email`, `name`, `currency`, `locale`, `timezone`
  - 응답 예: `{ id, email, name, currency, locale, timezone }`
- **PATCH /api/user** (또는 `/api/me`)
  - body: `name?`, `currency?`, `locale?`, `timezone?`
  - 데모 유저 정보 일부 수정

(추후 로그인 도입 시 “현재 로그인한 유저”로만 접근하도록 변경)

### 3-2. 프론트엔드
- **설정 페이지** (또는 “프로필” 섹션)
  - “프로필” / “내 정보”: 이름, 이메일(표시만), 통화, 로케일, 타임존 표시 및 수정
  - `GET /api/user` 로 표시, `PATCH /api/user` 로 저장
- 필요 시 **상단/사이드에 현재 유저 이름** 표시 (선택)

이렇게 하면 **User가 화면에 반영**되고, “User 관리”는 “현재 1명 프로필 관리” 수준으로 가능해짐.

---

## 4. 로그인 필요 여부와 단계

- **지금**: User 1명(데모) 고정 → 로그인 없이도 “현재 유저 프로필”만 보여주면 됨.
- **나중에 “여러 사람이 각자 로그인”**을 원하면:
  - 로그인(이메일+비밀번호 또는 OAuth 등)
  - 세션 또는 JWT로 “현재 userId” 결정
  - 기존 `getDemoUserId()` 대신 “현재 로그인한 userId” 사용

**권장 단계**
1. **1단계(우선)**: 로그인 없이 **GET/PATCH /api/user** + 설정 화면에서 User(프로필) 조회·수정만 반영.
2. **2단계(선택)**: 비밀번호 필드 추가·로그인 API·로그인 화면·세션 또는 JWT 도입.

즉, “User가 로그인할 때 쓰는 그 User”라면 **로그인 기능이 필요**하고, 당장은 **프로필만 화면에 반영**해 두고, 로그인은 별도 단계로 설계하는 것이 무난함.

## 5. 상위-하위 삭제 정책 (연관 데이터 유지)

요구: **지출/수입 이력을 지우지 않고**, 상위(카테고리·계정 등) 삭제 시 정책을 정함.

### 5-1. 연관 관계 정리

| 상위(삭제 대상) | 하위(연관 데이터) | 현재 스키마/동작 |
|-----------------|-------------------|------------------|
| User | Account, Category, Transaction, Budget | User 삭제 시 Cascade → 전부 삭제. (일반적으로 “유저 삭제”는 서비스 정책으로 제한) |
| Account | Transaction | FK만 있음(onDelete 미지정 → Restrict). **이미 409**로 “거래 있는 계정 삭제 불가” 처리됨. |
| Category | Transaction, Budget | FK만 있음. **이미 409**로 “거래/예산에서 사용 중이면 삭제 불가” 처리됨. |
| Category (parent) | Category (children) | parentId nullable. 부모 삭제 시 자식의 parentId를 **null**로 두는 선택 가능. |

이력 보존이 중요하므로: **Transaction은 삭제하지 않고**, “연결만 바꾼다(null 또는 다른 상위로 재할당)”가 맞음.

### 5-2. 정책 옵션

**A. nullable로 두기**
- 예: 카테고리 삭제 시, 해당 카테고리를 쓰는 **거래의 categoryId를 null**로 변경 후 카테고리 삭제.
- 장점: 삭제 단순.  
- 단점: `categoryId` null인 거래가 생겨, 목록/집계에서 “미분류” 등 별도 처리 필요.

**B. 다른 항목으로 재할당 후 삭제**
- 예: 카테고리 삭제 시, “이 카테고리를 쓰는 거래 N건을 **다른 카테고리로 모두 바꿀까요?**” → 사용자가 대체 카테고리 선택 → 일괄 update 후 삭제.
- 장점: 이력 유지하면서 연관이 깨지지 않음.  
- 단점: API·UI에서 “대체 대상 선택” 플로우 필요.

**C. 삭제 거부(현재 방식 유지)**
- “사용 중인 카테고리/계정은 삭제할 수 없습니다”만 안내.
- 장점: 구현 이미 됨.  
- 단점: 사용자가 “정말 안 쓸 카테고리인데 거래가 있어서 못 지운다”는 불편 감수.

### 5-3. 권장 방향

- **계정(Account)**  
  - **현재처럼 409 유지**해도 무방.  
  - 나중에 “이 계정의 거래를 모두 **다른 계정으로 이전**한 뒤 삭제” 기능을 넣을 수 있음 (B와 동일 사상).

- **카테고리(Category)**  
  - **B(재할당)** 우선 권장:  
    - 삭제 시 “이 카테고리를 사용하는 거래 N건, 예산 M건이 있습니다. **다른 카테고리로 모두 변경**한 뒤 삭제할까요?”  
    - 대체 카테고리 선택(드롭다운) → 확인 시  
      - 해당 카테고리를 쓰는 **Transaction**의 categoryId를 선택한 카테고리로 update  
      - 해당 카테고리를 쓰는 **Budget**의 categoryId를 선택한 카테고리로 update (또는 예산은 삭제할지 정책 결정)  
      - 그 다음 Category 삭제  
  - **A(nullable)** 는 “미분류”를 허용할 때만:  
    - categoryId nullable로 스키마 변경 + “미분류” 표시/필터 로직 필요.

- **카테고리 계층(parentId)**  
  - 부모 카테고리 삭제 시: 자식의 **parentId를 null**로 두면 “최상위 카테고리”로 남김.  
  - 또는 “자식이 있으면 삭제 불가”로 두고, 사용자에게 “자식 먼저 정리하세요” 안내.

정리: **이력(거래)은 지우지 않고**,  
- **삭제 거부(409) + “다른 항목으로 바꾼 뒤 삭제할래?” 안내 + (선택) 재할당 API/UI**  
로 가는 것이 요구사항(“이력 사라지면 안 됨”)에 맞음.

## 6. 구현 순서 제안

### Phase 1: User 화면 반영 (로그인 없이)
1. **서버**: `GET /api/user`, `PATCH /api/user` 추가 (데모 유저 기준).
2. **클라이언트**:  
   - user API·타입·스토어(또는 기존 설정 스토어에 현재 유저 필드) 추가.  
   - 설정(또는 프로필) 화면에서 이름·통화·로케일·타임존 표시 및 수정.
3. (선택) 헤더/레이아웃에 “현재 유저 이름” 표시.

### Phase 2: 삭제 정책·UX 정리
1. **문서화**:  
   - 카테고리/계정 삭제 시 “재할당 후 삭제” 플로우를 PRD/플로우에 명시.  
   - nullable 허용 여부(미분류) 결정.
2. **카테고리 삭제 개선**:  
   - 삭제 시도 시 안내 문구: “이 카테고리를 사용하는 거래 N건, 예산 M건이 있습니다. 다른 카테고리로 모두 변경한 뒤 삭제할까요?”  
   - 대체 카테고리 선택 + “변경 후 삭제” API (또는 기존 DELETE에 body로 대체 categoryId 받기).  
   - 프론트: 삭제 확인 다이얼로그에 위 문구 + 카테고리 드롭다운 + 확인 버튼.
3. **계정 삭제**:  
   - 당장은 409 유지.  
   - 필요 시 “이 계정의 거래를 모두 다른 계정으로 변경한 뒤 삭제” API·UI 추가 (카테고리와 유사 패턴).

### Phase 3 (선택): 로그인
1. User에 비밀번호(또는 OAuth 식별자) 관련 필드/테이블 설계.  
2. 로그인·로그아웃 API, 세션 또는 JWT.  
3. 로그인 화면, 라우트 가드(미로그인 시 로그인 페이지로).  
4. 기존 API에서 `getDemoUserId()` 대신 “현재 로그인 userId” 사용하도록 전환.

## 7. 검토 포인트

- **User 화면 반영**: GET/PATCH `/api/user` + 설정(프로필) UI로 “DB의 User가 화면에 반영”되는지 확인하면 됨.
- **User = 로그인 유저**: 맞다. 따라서 **실제 다중 사용자 서비스**를 하려면 **로그인 기능이 필요**하고, 1단계는 “한 명 프로필만 보이기”, 2단계에서 로그인 도입이 자연스러움.
- **삭제 정책**:  
  - “다 지우면 이력 사라짐” → **그렇게 하지 않음** (현재도 409로 막혀 있음).  
  - “null로 두기” vs “다른 목록으로 교체할래?” → **다른 목록(카테고리/계정)으로 교체**하는 플로우를 권장하고, 안내 문구와 대체 선택 기능을 넣는 방향이 요구사항과 일치함.

이 계획대로면 User 관리가 화면에 반영되고, 로그인은 단계적으로 도입할 수 있으며, 삭제 시 이력은 유지하면서 “재할당 후 삭제”로 정리할 수 있음.
